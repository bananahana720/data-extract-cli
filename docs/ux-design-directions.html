Data Extraction Tool - CLI Interaction Pattern Exploration

Chosen Direction: Git-Style Subcommands + Rich Feedback

Based on the brainstorming session analysis, we've selected a Git-style subcommand structure combined with Rich terminal output. This approach provides familiar patterns for developers while enabling the rich visual feedback and learning features that differentiate this tool.

Explored Design Directions

1. Git-Style Subcommands Chosen

Familiar pattern used by git, docker, npm. Main command with subcommands for different operations. Most developers already know this pattern instinctively.

$ data-extract process ./docs/ --output ./clean/
$ data-extract analyze ./corpus/ --report
$ data-extract validate ./input/
$ data-extract config set output.format json
$ data-extract cache clear
Strengths
  • Universally familiar pattern
  • Easy to extend with new commands
  • Clear separation of concerns
  • Great tab completion support
Considerations
  • Requires learning command names
  • Can be verbose for simple tasks
2. Interactive Guided Mode Integrated

When no arguments provided, the tool asks questions to guide the user. Perfect for novices and complex configurations. Integrated as fallback mode.

$ data-extract
Welcome to Data Extraction Tool!
? What would you like to do?
▶ Process documents
Analyze corpus
Configure settings
View help & tutorials
? Select files to process:
☑ quarterly-report.pdf
☑ audit-findings.docx
☐ old-draft.pdf
Strengths
  • Zero memorization required
  • Discoverability built-in
  • Self-teaching through use
Considerations
  • Slower for power users
  • Not scriptable
3. Config-Driven Pipeline Supported

Define pipelines in YAML/JSON configuration files. Execute by referencing config. Great for repeatable workflows and CI/CD integration.

# pipeline.yaml
input: ./documents/*.pdf
output: ./clean/
stages:
- extract
- normalize
- chunk:
size: 500
- semantic
$ data-extract run --config pipeline.yaml
Strengths
  • Fully reproducible
  • Version controllable
  • CI/CD friendly
Considerations
  • Setup overhead for one-offs
  • YAML learning curve
4. Unix Pipeline Style Alternative

Compose operations using Unix pipes. Each stage is a separate command. Maximum flexibility but requires understanding data flow.

$ cat report.pdf | data-extract extract | data-extract normalize
$ find . -name "*.pdf" | \
data-extract extract --batch | \
data-extract analyze --json | \
jq '.duplicates'
Strengths
  • Unix philosophy
  • Composable with other tools
  • Stream processing
Considerations
  • Complex for beginners
  • Error handling tricky
  • No rich output in pipes
5. Make-Style Targets Future

Define targets that represent outcomes. Tool figures out what needs to run. Dependency-aware, only runs what's needed.

$ data-extract all
Running: extract -> normalize -> chunk -> semantic -> output
$ data-extract clean
Removing cached models and intermediate files...
$ data-extract semantic
extract, normalize, chunk up-to-date
Running: semantic
Strengths
  • Dependency tracking
  • Incremental processing
  • Simple common cases
Considerations
  • Hidden complexity
  • Less explicit control
6. TUI Dashboard Mode Roadmap

Full terminal UI application using Textual. Multiple panes, real-time updates, keyboard navigation. The "Hobbyist mode" visualization platform.

┌─ Files ───────┐ ┌─ Pipeline ───────┐ ┌─ Quality ───────┐

│ ☑ audit.docx │ │ ■ Normalize │ │ │

[Tab] Switch pane [Enter] Select [Space] Toggle [Q] Quit
Strengths
  • Visual, interactive
  • Real-time updates
  • No context switching
Considerations
  • Complex to build
  • Not scriptable
  • Terminal compatibility
